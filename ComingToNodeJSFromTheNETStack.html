A basic NodeJS project

Just as an example, here is a basic project that takes a request URL and writes it back with a response to the page. 

var http = require("http");

var server = http.createServer(function(req, res){
console.log(req.url);
resp.write("" + req.url + "");
resp.end();
});

server.listen(3000);

Essentially what you're doing in the above code block is establishing a server that takes in a request and provides a response. You also specify the port that you will be listening on, in our case it is "3000"

Express.js

Express.js is a web framework for NodeJs to take on views, routing, database handling and everything else you would expect from a web framework. It is basically a requirement that you include it in your Node.js project if you're developing a web application. TLDR: Get it.

NPM

Stands for "Node Package Manager". Think of NuGet except the search tools aren't as refined. The UI is basically non-existent, no matter what IDE you use, but that's alright because the command line is perfectly fine for grabbing packages. An example of a command line would be: 

npm install express --save
npm install underscore --save
npm install vash --save
npm install jade --save
npm uninstall jade --save

    Thoughts on Node.js

    I'll be honest, I haven't truly mastered Node.js nor spent months developing in it but I have a few complaints that made me focus my learning efforts in other directions. Here is my list of "complaints":

    • IDE: Since Visual Studio doesn't natively support Node.js with rich intellisense and all of the other fun features that are available with .NET, it is very difficult to find an IDE that "just works". The IDE either makes your life easy or painful. In the case of Node, Visual Studio made it painful. I used Webstorm halfway through the project, but it was basically the same experience with a nicer UI.
    • Javascript: When you're working with C# for your backend needs, it just feels so natural. Javascript just doesn't feel very natural as a backend language interfacing with databases and feeding code to the view. On top of Javascript not being a true Object Oriented Language(not without some clever tactics), it just feels like I have to constantly review my code to make sure I didn't make a typo somewhere. Sure some of this could be resolved by the IDE, but where is the perfect IDE solution?
    • NoSQL and MongoDB. Yeah...document databases...they're great for searching documents I guess. When you're coming from a rich interface and familiarity with SQL, it's not a simple transition going to Node by any means. MongoDB is very lightweight and "simple" to get started but I have a hard time replacing SQL. That said, there are drivers that interface with SQL, but as nice as you are probably familiar with in .NET. If you're looking for a good driver to interface with SQL Server on your Node.js project, I hear great things about sql-cli. You can install it via npm "npm install sql-cli --save"
    • The Framework: The .NET framework really does a lot for the developer and shields them from the obscure details that do not generate content but basically "setup" the framework. Dabbling with routes and getting into the minutia of how to feed data to the page gets old really fast. Some people may like the "pure" feel of Node.js, but I find it a chore.
    • Documentation: There is a lot of good documentation out there but nothing matches .NET. I spent quite a bit of time looking around for the right api to use.
    • Debugging: Debugging is not as you would expect, mostly because the IDE's don't support it as you would expect. I could write a whole story about Javascript debugging but I will leave it at being ineffective for Node.js applications.
